## Problem Statement

Developer configs, dotfiles, hidden settings, and environment setups are scattered across multiple machines with no unified way to see, compare, sync, or share them. Existing tools each solve one slice — Syncthing syncs files, chezmoi manages dotfiles, Nix reproduces environments, Tailscale connects machines — but none combine them into a single GUI experience. Every tool requires terminal fluency. The result: developers manually email files, create throwaway git repos, or just accept that their machines drift apart.

The specific trigger: Claude Code settings in `~/.claude/` on one machine that should also be on another, but the process of getting them there (git init, push, clone, deal with gitignored hidden files) is absurd for what should be a drag-and-drop operation.

## What Makes This Cool

Three things nobody else is doing together:

1. **Unified device browser** — see all your machines' filesystems in one app, like Gmail's multi-account inbox. Browse, search, drag-drop between devices. Zero terminal.

2. **Machine Diff** — select two devices and see not just which files differ, but HOW they differ at four levels: presence (file exists/missing), metadata (size/date), content (line-by-line diff), and semantic (environment-level: "Machine A has Python 3.12 via pyenv, Machine B has 3.11 via Homebrew"). Spatial graph visualization shows your device ecosystem and where machines converge/diverge.

3. **Capability packaging** — take a complex multi-step setup (Python ML environment, shell customization, dev toolchain) and package it into a shareable unit that someone else can apply with one click. Not a Dockerfile (too heavy for one file), not a Nix flake (too complex). Tiered packaging that auto-detects the right level of complexity.

The insight: file sync, environment management, and remote access are the same problem at different zoom levels. A single UI handles the full spectrum.

## Constraints

- Must feel like Finder/AirDrop/Gmail — zero terminal requirement for basic operations
- Power users get an embedded smart terminal (context-aware, knows which device you're targeting)
- Desktop app (Tauri v2) — needs local filesystem access, background sync, SSH tunneling
- Fully peer-to-peer, no cloud dependency, open source
- Cross-platform: macOS first, Linux second, Windows third
- The sync layer handles both individual files and named packages from day one

## Premises

1. **GUI composition layer over existing infra** — not building new sync or networking primitives. iroh (P2P/QUIC) or Tailscale for networking, rsync/SFTP for transfer, SSH for remote access.
2. **MVP = link two devices + browse + drag-drop copy** — everything else layers on top of this working correctly.
3. **Desktop app via Tauri v2** — not Electron (150MB), not web (can't do local FS access reliably). Tauri gives ~15MB binary with Rust backend.
4. **Sync layer's data model supports named packages from day one** — the architecture treats "sync one file" and "sync a capability pack" as the same operation at different granularities. The packaging UI ships incrementally, but the underlying data model never needs migration.
5. **Open source, peer-to-peer, no cloud dependency** — distributable via GitHub Releases, Homebrew, apt.

## Approaches Considered

### Approach A: "Finder++" — File-First, Capability Later
- Build dual-pane file browser with P2P networking first
- Add tiered packaging as second milestone
- Tauri v2 + React + iroh
- Effort: M (2-3 weeks browser, 3-4 weeks packaging)
- Risk: Low. Proven stack, clear separation.
- Pro: Fastest to something usable. Weekend prototype possible.
- Con: The "whoa" moment (graph viz, Machine Diff) comes later.

### Approach B: "Capability Manager" — Package-First Architecture
- Design around capability pack format from day one
- File browsing is a view INTO packages
- Tauri v2 + React + Tailscale
- Effort: L (4-6 weeks to MVP)
- Risk: Medium. Pack format design could stall progress.
- Pro: Differentiator leads. Tailscale handles auth/ACLs.
- Con: Slower to first demo. Tailscale account required.

### Approach C: "Machine Diff" — Graph-First, Environment-Aware
- Start with environment agent that inventories each machine
- Hero screen is spatial diff between two machines
- Browsing and packaging emerge from the graph
- Tauri v2 + React + d3-force + iroh
- Effort: L-XL (6-8 weeks)
- Risk: High. Most novel, most likely to get stuck.
- Pro: Nobody else does this. Strongest long-term moat.
- Con: Environment scanning is OS-specific and fragile.

## Recommended Approach

**Hybrid A+C: File browser foundation with Machine Diff as a core feature (not stretch goal).**

Uses iroh (not Tailscale) for P2P transport — see rationale in Key Technical Decisions below.

**Timeline risk note:** Estimates assume some familiarity with Tauri v2 and iroh. If this is first contact with either, add 1-2 weeks of ramp-up. The CLI prototype in Week 1 is specifically a risk-reduction exercise — if iroh P2P pairing does not work reliably by end of Week 1, reassess the transport layer.

Build order:
1. **Week 1-2: Core shell + P2P pairing** — Tauri v2 app, iroh-based device pairing (QR code or key paste), dual-pane file browser with local + remote filesystem listing.
2. **Week 2-3: Drag-drop sync + content diff** — File/folder transfer via iroh-blobs, side-by-side content diff (monaco-editor), multi-level diff indicators in the browser (green/yellow/red).
3. **Week 3-5: Machine Diff + environment agent** — Lightweight daemon on each machine that inventories installed tools, detects package managers, builds dependency graph. Spatial graph visualization (d3-force) showing device ecosystem. "Diff these two machines" as a first-class view.
4. **Week 5-7: Tiered packaging** — Capability pack format (see below). Auto-detect packaging tier. Select files/tools in UI, name the pack, sync or share.
5. **Week 7+: Smart terminal** — Embedded context-aware terminal.

### Future Vision (explicitly out of scope for initial release)
- **AI-powered config intelligence** — auto-detect what matters, suggest syncs, explain configs
- **Environment marketplace** — browse and share curated capability packs publicly
- **Tier 4 environment snapshots** — generating Dockerfiles/Nix flakes from env scans is a research problem, not an engineering task. Deferred until Tiers 1-3 are proven.

### Tiered Packaging System

The packaging system auto-detects complexity based on what the user selects:

| Tier | Name | What it transfers | Format | Example |
|------|------|-------------------|--------|---------|
| 1 | File Bundle | Raw files | Tar/zip with manifest | Copy .claude/ directory |
| 2 | Config Package | Files + placement rules + templating | TOML manifest + files | .zshrc with machine-specific PATH templating |
| 3 | Capability Pack | Files + dependency declarations + install steps | TOML manifest + files + scripts | Python ML setup: pyenv + Python 3.12 + virtualenv + requirements.txt |
| 4 | Environment Snapshot (Future) | Full dependency graph export | Generated Dockerfile or Nix flake | Complete dev environment replication |

Detection logic:
- User selects only files with no external dependencies -> Tier 1
- Files reference machine-specific paths or env vars -> Tier 2
- Files depend on tools/runtimes not present on target -> Tier 3
- User explicitly requests "clone this machine's setup" -> Tier 4

The user never sees tier numbers. They just select what to sync, the app figures out what's needed, and shows a preview: "To apply this on Machine B, I'll need to: install pyenv, set up Python 3.12, create a virtualenv, install 47 packages. Proceed?"

### Multi-Level Diff System

| Level | When shown | Cost | What it tells you |
|-------|-----------|------|-------------------|
| Presence | Always (in browser) | Free — directory listing | "Machine A has this, Machine B doesn't" |
| Metadata | Always (in browser) | Cheap — stat call | "Both have .zshrc, different size/date" |
| Content | On click, or auto for known configs | Medium — read both files | "Line 47 differs: PATH includes pyenv on A" |
| Semantic | Environment agent, recognized configs | Expensive — parsing | "Machine A has oh-my-zsh with 3 plugins, B has vanilla zsh" |

Color coding in file browser: green (identical), yellow (differs), red (missing on one side). Click yellow for content diff. Known config files (.zshrc, .gitconfig, .claude/, .ssh/config) auto-expand content diff.

### File Browser Actions

Action paths available in the file browser:
- **Context menu (right-click):** Open, Open With..., Copy Path, Reveal in Finder, Copy to [device], Delete, Rename, Get Info
- **Drag-and-drop:** Drag file(s) to device in sidebar to copy
- **Toolbar:** "Send to..." dropdown with paired device list
- **Keyboard:** Cmd+C to copy reference, Cmd+V on another device's browser to paste
- **Multi-select:** Shift+click range, Cmd+click individual, then any action applies to all

### Diff View Actions

Actions in the Machine Diff view:
- **Per-row:** ← → directional sync arrows to copy in either direction
- **Missing files:** inline "Copy to [device] →" button
- **Bulk:** "Sync All →" / "← Sync All" buttons in toolbar
- **Checkbox select:** Select multiple rows + "Sync Selected →"
- **Right-click:** Copy to Left, Copy to Right, Ignore, Add to Package

**Offline device in diff:** Shows cached directory listing with amber banner "Showing cached data from 2h ago. Sync unavailable until device reconnects." All cached views are read-only.

### Key Technical Decisions

**iroh over Tailscale for P2P transport:**
- iroh embeds as a Rust library directly into Tauri backend — no separate daemon, no account signup
- Ed25519 keypair generated on first launch, NAT traversal handled automatically
- Content-addressed blob transfer (iroh-blobs) gives resumable, verified transfers
- Trade-off: smaller community than Tailscale, less battle-tested at enterprise scale
- Decision: start with iroh for zero-dependency setup. Tailscale support as optional backend later.
- **Risk mitigation:** Abstract the transport layer behind a trait (`TransportBackend`) from day one. iroh is the first implementation. If iroh proves unreliable, the same trait can be implemented over SSH/SFTP or Tailscale with minimal frontend changes. This is a 1-2 day investment that eliminates single-vendor risk.

**Tauri v2 over Electron:**
- ~15MB binary vs ~150MB
- Rust backend is natural fit for iroh integration
- Better security model (explicit capability permissions)
- Trade-off: smaller ecosystem than Electron, some Web APIs unavailable
- Decision: Tauri. The size and performance difference matters for a tool people install on every machine.

**SQLite for local state:**
- Device graph, package metadata, sync history, environment inventory
- Each machine maintains its own SQLite DB. DBs are never synced as raw files (corruption risk). Instead, structured metadata (device info, environment graph summaries) is exchanged as RPC messages, and each machine rebuilds its own view from those messages.
- No external database dependency

**Environment agent as separate lightweight daemon:**
- Runs on each linked machine
- **V1 scope (macOS):** detects shell + framework (bash/zsh, oh-my-zsh/starship), Homebrew packages, explicitly listed version managers (pyenv, nvm, rbenv). All other tools detected by PATH scanning only (name + version from `--version`). Deep integration for additional tools added incrementally based on user demand.
- **Windows note:** WSL instances are treated as separate devices from the Windows host. Windows agent detects winget/scoop/chocolatey packages. Full WSL support in a later release.
- Builds local dependency graph (adjacency list in SQLite)
- Watches for changes (filesystem events) and updates graph
- Communicates via RPC over iroh bi-directional streams. The Tauri app embeds the agent for the local machine; remote machines run the agent as a standalone daemon. RPC is only used for cross-machine communication.

**Agent RPC protocol (v1):**
```
list_dir(path) -> Vec<Entry>
read_file(path, range?) -> Bytes
write_file(path, bytes) -> Result<()>
write_dir(path, entries) -> Result<()>
stat(path) -> Metadata
diff_content(path_local, path_remote) -> DiffResult
apply_package(manifest) -> Result<ApplyReport>
inventory() -> EnvironmentGraph
```

**Security note:** All RPC calls are authenticated via the iroh Ed25519 keypair established during device pairing. Only paired devices can issue RPC commands. Write operations (`write_file`, `write_dir`, `apply_package`) require explicit per-session user confirmation on the receiving device for V1. Fine-grained capability scoping is handled by the Permission Model below (3-tier system with directory scoping).

### Permission Model

3-tier permission system for paired devices:

| Tier | Name | Capabilities | Default |
|------|------|-------------|---------|
| 1 | Browse Only | See file listings, metadata, diffs | Default for new pairings |
| 2 | Read + Copy | Tier 1 + copy files FROM this machine | Explicit upgrade |
| 3 | Full Access | Tier 2 + write/modify files ON this machine | Explicit upgrade + per-action confirmation |

Key rules:
- Changes always apply to ONE machine (directional, not bidirectional sync)
- Tier 3 write operations show a native OS confirmation on the receiving machine: "Mac Mini wants to write .zshrc. [Allow Once] [Allow Always for This Session] [Deny]"
- Directory scoping: restrict access to specific directories within any tier
- Instant revocation from Settings — remote device loses all access immediately
- Permission tier visible as badge in sidebar: [Full] green, [Read+Copy] orange, [Browse] blue

**Permission Grant data model:**
```
PermissionGrant {
  peer_id: Ed25519PublicKey,
  tier: Browse | ReadCopy | Full,
  paths: Vec<PathScope>,      // empty = everything visible
  expires_at: Option<u64>,    // unix timestamp, for temporary sharing
  label: String,              // "personal", "work", "alice-collab"
}
```

Enforcement is server-side at the RPC layer. Every `list_dir`, `read_file`, `write_file` call checks the peer's PermissionGrant: is this peer paired? Is this path in scope? Does their tier allow this operation? If any check fails, return `PermissionDenied`. The UI never constructs a request it can't make, but enforcement lives at the daemon regardless.

**Labeled sidebar groups (not multi-network):** The `label` field on PermissionGrant enables client-side grouping in the sidebar without any protocol-level network concept. Labels are local-only. No workspace switching. No network overlay. Devices are grouped by label:
```
  Personal
    Mac Mini          [online]  [Full]
    ThinkPad          [offline] [Full]
  Work
    staging-server    [online]  [Read]
    alice-macbook     [online]  [Browse]
  Temp
    bob-laptop        [expires in 22h]  [Read · ~/.claude/ only]
```

This solves personal, work, collaboration, and temporary sharing scenarios without the complexity of multiple networks. The decision: per-device permission scoping, not separate meshes. Inspired by Tailscale's pivot from multiple tailnets to ACL grants.

**Connectivity and offline behavior:**
- iroh includes relay nodes for NAT traversal fallback. If direct P2P fails (symmetric NAT, corporate firewall), traffic routes through iroh relay servers automatically.
- When a remote machine is unreachable: device shows "offline" badge with last-seen timestamp. The app caches the most recent directory listing and environment graph, displayed with a staleness indicator ("last synced 3 hours ago"). Cached views are read-only — no sync/copy operations until connection restores.
- Large file transfers: progress bar with pause/resume (iroh-blobs supports resumable). Warning prompt above 500MB ("This transfer is 2.3 GB. Proceed?").

### Offline Transfer Queue

iMessage-pattern queued transfers:
- When a user copies a file to an offline device, DevMesh queues the intent
- UI shows: "ThinkPad is offline. Queue transfer? It will complete automatically when ThinkPad connects."
- SQLite table: `pending_transfers(id, source_device, target_device, source_path, target_path, created_at, status, completed_at)`
- On reconnect: source device pushes queued blobs via iroh. Target device shows consent prompt.
- Notification on completion: "Transfer to ThinkPad complete: 3 files synced."
- Badge on device icon shows pending transfer count
- Queued operations respect the same permission tier as live operations

**Future (Horizon 2): Watched paths with diff-then-confirm.** User pins specific files as "keep identical." On reconnect, shows diff + one-click apply. Not auto-sync — a watched diff with manual trigger.

**Device onboarding:**
Both machines must have DevMesh installed. The agent daemon auto-installs on first launch. There is no "browse a machine that doesn't have the app" flow — both ends need the app running. A lightweight agent-only install (no GUI, CLI pairing) is a future option for headless servers.

### Device Pairing UX

Two-tier pairing system that replaces the original ticket-paste approach:

**Tier 1: Same-Network Auto-Discovery (mDNS)**
- Both devices on the same WiFi are discovered automatically via mDNS/Bonjour
- Service type: `_devmesh._tcp.local.`
- Zero user input required. Device appears in the pairing sheet within 2-3 seconds.
- Both sides see a mutual approval dialog. Click "Allow" on both. Done.
- Uses `mdns-sd` crate (pure Rust, no system deps)

**Tier 2: Short PIN via Relay Rendezvous**
- For cross-network pairing (different WiFi, home + office)
- 6-digit PIN displayed as "482 · 916" (two groups of 3)
- PIN is ephemeral (5-min TTL), single-use, registered with iroh relay
- Device B enters PIN, relay matches them, connection established
- No long ticket strings, no QR codes needed

The user never thinks about which tier to use — the app tries mDNS first, and if the other device isn't found locally within a few seconds, it offers the PIN flow.

**Semantic diff extensibility:**
Semantic diff ships with built-in parsers for: `.zshrc`/`.bashrc` (detect frameworks, plugins, PATH entries), `.gitconfig` (sections), `package.json` (dependencies), `Brewfile` (packages). Parser interface is pluggable via a `DiffParser` trait — community can contribute parsers for new config types.

### Graph Visualization Strategy

Graphs used in exactly two places:

**1. Capability Pack Dependency DAG**
- Directed acyclic graph showing install dependencies for Tier 3 packs
- Layout: dagre or elkjs (deterministic directed layout, NOT force-directed)
- Interaction: hover node for details, click to expand sub-dependencies, red badges on failing nodes
- Scalability: collapsible groups for large dependency sets (e.g., "47 packages" as one node)
- Library: react-force-graph for canvas rendering, dagre for layout computation

**2. Device Topology Widget**
- Small floating sidebar widget showing paired device mesh (3-8 nodes)
- Click node to navigate to that device
- NOT a hero/full-screen view (too few nodes to justify the space)

**NOT built as graphs:** environment dependency browsing (searchable tree view instead), config file relationships (indented list), diff view device indicator (header text, not floating widget).

## Open Questions

1. **Package format spec** — TOML manifest is the working assumption, but needs detailed design. What fields? How do install steps handle OS differences? How do rollbacks work?
2. **Conflict resolution for config files** — "Newer wins" is too simple for configs that get independently edited on both machines. Semantic merge (by section for .gitconfig, by block for .zshrc) is ideal but complex. What's the MVP conflict UX?
3. **Security model for device pairing** — iroh uses Ed25519 keys, but what's the consent flow? QR code scan? Manual key verification? How do you revoke a linked device?
4. **Environment agent scope** — how deep does the inventory go? Just installed CLI tools + configs? Or also GUI apps, fonts, system preferences? The deeper you go, the more OS-specific fragility.
5. **Smart terminal architecture** — xterm.js frontend + pty backend in Rust, but how does it know context? Does it read the current file browser path? Does it auto-suggest commands based on what you're looking at?
6. **Naming** — "DevMesh" is a working title. The product needs a name that communicates "your devices, unified" without sounding enterprise-y.
7. **Multi-network mesh** — RESOLVED: Don't build separate networks. Use per-device PermissionGrant with client-side labels for sidebar grouping. Solves personal/work/collaboration/temporary scenarios without protocol complexity. See Permission Model section.

## Success Criteria

- **Week 3 milestone:** Two machines paired, browsing each other's filesystems, drag-drop copy works. Feels like magic.
- **Week 5 milestone:** Machine Diff view shows environment-level differences between two machines. Spatial graph renders device ecosystem.
- **Week 7 milestone:** Tier 1-3 packaging works. Can package a Python dev setup on Machine A and apply it on Machine B with one click.

- **Open source traction:** 100+ GitHub stars within first month of public release. At least 5 community-contributed capability packs.

## Distribution Plan

- **Primary:** GitHub Releases — pre-built binaries for macOS (universal), Linux (amd64, arm64), Windows (amd64)
- **Package managers:** Homebrew tap (`brew install devmesh`), AUR for Arch Linux, potential Winget/Scoop for Windows
- **CI/CD:** GitHub Actions. Tag-based releases. Build matrix for all platforms. Code signing for macOS (notarization required for Gatekeeper).
- **Auto-update:** Tauri's built-in updater (checks GitHub Releases for new versions)
- **Agent daemon:** Bundled with the app. Installs as a launch agent (macOS) / systemd service (Linux) / Windows service on first "link this device" action.

## Reviewer Concerns

These items were flagged by adversarial review and remain as watch-items rather than blocking issues:

1. **Agent timeline compression:** Weeks 4-5 combines the environment agent daemon + filesystem watching + semantic diff parsers. Consider cutting filesystem watching from V1 (scan-on-demand instead of live-updating) or moving semantic diff parsers to Week 6 alongside packaging work, which benefits from the same structured config understanding.
2. **Fine-grained RPC security:** V1 uses device-level auth (paired = full access). Directory-scoped permissions, read-only pairing, and per-operation consent UX need design before any multi-user or team deployment.
